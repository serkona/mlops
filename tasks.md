# MLOps-пайплайн для классификации мошеннических транзакций

## Описание проекта
**Задача:** Реализовать полный цикл разработки и внедрения модели машинного обучения для классификации мошеннических транзакций. Проект представляет собой единый, последовательно развивающийся репозиторий, охватывающий все этапы жизненного цикла ML-системы.

**Источник данных:** [Credit Card Fraud Detection](https://www.openml.org/search?type=data&id=43627&sort=runs&status=active) — классический датасет с сильным дисбалансом классов (мошеннические транзакции составляют малую долю от общего числа), что идеально подходит для имитации реальных задач.

**Стек технологий:**
- **Язык:** Python
- **Контейнеризация:** Docker
- **Версионирование данных:** DVC
- **Трекинг экспериментов:** MLflow
- **API:** FastAPI
- **Качество данных:** Great Expectations
- **CI/CD:** GitHub Actions

### Неделя 5: Unit-тесты и интеграционные тесты API

Цель: построить надёжный многоуровневый тестовый щит, который поймает регрессии в коде — до того, как они попадут в прод.

#### 1. Unit-тесты функций предобработки (`pytest`)

- Написать unit-тесты для **каждой** функции в `preprocess.py`.
- Использовать **параметризацию** (`@pytest.mark.parametrize`) для покрытия граничных случаев:
  - пустой DataFrame,
  - датасет из одного класса (все транзакции — мошеннические),
  - столбец с полностью пропущенными значениями (`NaN`),
  - данные с некорректными типами (строка вместо числа в `Amount`).
- Подключить **`pytest-cov`** и добиться покрытия кода тестами **≥ 80%**. Добавить в `pytest.ini` / `pyproject.toml` флаг `--cov-fail-under=80`, чтобы тесты падали автоматически при снижении покрытия.

#### 2. Интеграционные тесты API

- Написать интеграционные тесты с помощью **`pytest` + `httpx`** (или `TestClient` из FastAPI) для эндпоинта `/predict`:
  - **Happy path:** корректный JSON → HTTP 200 + поле `fraud_probability` типа `float` в диапазоне [0, 1].
  - **Негативные сценарии:**
    - отсутствует обязательное поле → HTTP 422,
    - передан неверный тип (`"Amount": "abc"`) → HTTP 422,
    - передан вектор неправильной размерности → HTTP 422 или 400 с информативным сообщением.
  - **Нагрузочный мини-тест:** отправить 100 последовательных запросов и убедиться, что медианное время ответа не превышает **200 мс** (использовать `time.perf_counter` в цикле).

> **Примечания:**
> - Для теста на неправильную размерность: Pydantic не проверяет количество элементов в `List[float]` автоматически. Нужно добавить явный `@validator` (или `model_validator`) в схему запроса, иначе запрос пройдёт валидацию и упадёт внутри модели с HTTP 500 вместо 422.

### Неделя 6: CI/CD пайплайны (GitHub Actions)

Цель: автоматизировать весь путь от коммита до готового образа, включая контроль качества модели и безопасность зависимостей.

#### 1. Основной Workflow: линтинг, тесты, покрытие

Создать файл `.github/workflows/ci.yml` со следующей структурой:

- **Матричная стратегия:** запускать пайплайн на **двух версиях Python** (например, 3.10 и 3.11) одновременно.
- **Кэширование зависимостей:** использовать `actions/cache` для кэширования `pip` / `poetry` окружения по хэшу `poetry.lock` / `requirements.txt` — повторный запуск должен занимать существенно меньше времени.
- **Шаги:**
  1. `flake8` — линтинг (нулевая толерантность к ошибкам).
  2. `pytest` с `--cov` — запуск тестов и сборка отчёта покрытия.
  3. Загрузка отчёта покрытия как артефакт с помощью `actions/upload-artifact`.
  4. **Проверка безопасности зависимостей:** запустить `pip-audit` — workflow должен упасть при обнаружении известных уязвимостей (CVE).


#### 2. Release Workflow: сборка и публикация Docker-образа

Создать файл `.github/workflows/release.yml`:

- Триггер: создание git-тега вида `v*.*.*` (семантическое версионирование).
- **Шаги:**
  1. Сборка Docker-образа приложения — использовать `Dockerfile` из недели 4 (сервинг API), не `Dockerfile` из недели 1 (dev-окружение).
  2. Тегирование образа двумя тегами: точная версия (например, `v1.2.3`) и `latest`.
  3. Публикация образа в **Docker Hub** (бесплатно для публичных образов, без ограничений по объёму).
  4. **Smoke-тест образа:** поднять контейнер (`docker run -d -p 8000:8000`), дождаться готовности сервиса, затем проверить эндпоинт `/predict`.
     - Ожидание готовности: `curl --retry 10 --retry-connrefused --retry-delay 2 http://localhost:8000/health`
     - Проверка: `curl --fail -X POST http://localhost:8000/predict ...`

> **Примечания:**
> - Эндпоинт `/health` (возвращает `{"status": "ok"}`, HTTP 200) должен быть добавлен в FastAPI сервис из недели 4 — без него шаг ожидания никогда не завершится.
> - Флаг `--fail` в `curl` обязателен: без него job будет зелёным даже при HTTP 500.
> - Не отправлять запрос сразу после `docker run` — использовать retry-цикл выше, иначе тест упадёт с connection refused.
